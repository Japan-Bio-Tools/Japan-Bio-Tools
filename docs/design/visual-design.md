# Visual Design

本ドキュメントは、Japan-Bio-Tools 全体および BioFile Guide for Structure の共通ビジュアル方針を定義する正本である。  
個別画面の詳細実装は各アプリの実装と個別設計に従うが、色、階層、タイポグラフィ、情報の重みづけ、リソース階層の扱いは本書を優先する。

- 対象:
  - Japan-Bio-Tools Portal
  - BioFile Guide for Structure
  - 将来追加される Japan-Bio-Tools 配下の各サブアプリ
- 非対象:
  - 判定ロジック
  - 契約 shape
  - next_links 選定ロジック
  - warning priority
  - adapter / application / domain の意味

---

## 1. ビジュアル原則

Japan-Bio-Tools のビジュアルは、次のブランド原則を表現する。

- 静かなる精密さ
- 日本的な抑制
- 科学的信頼
- 編集的な規律
- プライバシー優先
- 証拠第一
- 将来を見据えた拡張性

目指すべき姿は、派手なバイオテック系ダッシュボードでも、アメリカ型SaaSのマーケティングサイトでもない。  
ブラウザネイティブでありながら、落ち着き、規律、説明可能性を備えた「日本製の科学的読解・分析用の精密機器」のような佇まいを目指す。

---

## 2. デザインの基本姿勢

### 2.1 何を優先するか
最優先は、見た目の派手さではなく「読むべき情報の優先順位」が明快であること。

特に BioFile Guide では、次の順で読めることを最優先とする。

1. 何者か
2. 注意点
3. 次の一手

### 2.2 何を避けるか
以下の方向には寄せない。

- 巨大CTA中心のSaaSランディング
- ネオン、グロウ、強いグラデーション
- サイバーパンク風のバイオテック表現
- 情報を一度に全部見せるコントロールルーム型UI
- 過度なスコア表示やダッシュボード化
- 直接的な和風モチーフの多用

日本らしさは、モチーフではなく「余白」「規律」「抑制」で表現する。

---

## 3. カラースキーム

色は和色ベースの落ち着いたトーンを用い、フラットで抑制された印象を守る。  
過度な彩度や装飾効果は使わない。

| トークン名 | 色名 / 役割 | 用途 |
|---|---|---|
| `--color-primary` | Sumi Navy（墨紺） | 主要見出し、ブランド、強い権威性を示す箇所 |
| `--color-secondary` | Iron Blue-Gray（鉄紺 / ブルーグレー） | サブ見出し、補助領域、安定感のあるUI要素 |
| `--color-bg` | Mineral White（白緑 / オフホワイト） | 全体背景 |
| `--color-surface` | Surface White | カード、パネル、入力面 |
| `--color-border` | Soft Border | 境界線、区切り |
| `--color-text` | Main Text | 基本文字色 |
| `--color-text-muted` | Muted Text | 補助説明、英語ラベル、注記 |
| `--color-accent-trust` | Aizome Cyan（藍染シアン） | Verified、推奨アクション、信頼できる導線 |
| `--color-accent-caution` | Restrained Amber（抑制された琥珀） | 注意喚起、控えめな警告、スポンサー明示 |
| `--color-accent-error` | Deep Red（深い赤） | 致命的エラー、無効状態のみに限定使用 |

### 3.1 使用ルール
- 原則として背景は明るく保つ
- Primary は広く塗りつぶしすぎない
- Cyan は「推奨」「信頼」の限定用途に留める
- Amber は注意とスポンサー表示に限定し、危険色としては使いすぎない
- Red は致命的状態のみに使う

---

## 4. タイポグラフィ

### 4.1 基本方針
日本語を主役とし、英語は補助ラベルとして扱う。  
英語は飾りではなく、補助的な意味付けに留める。

### 4.2 フォント方針
- Sans:
  - UI本文、一般ラベル、フォーム、本文説明
  - 例: Noto Sans JP
- Serif:
  - ブランド名、主要見出し、セクションタイトル
  - 例: Noto Serif JP
- Monospace:
  - PDB ID、拡張PDB ID、技術値、構造化値
  - 乱用しない

### 4.3 階層
最低限、次の階層を持つ。

1. ブランド見出し
2. 画面主見出し
3. セクション見出し
4. 補助英語ラベル
5. 本文
6. 技術値
7. バッジ / メタラベル

### 4.4 組版
- 詰め込みすぎない
- 行間と余白で読ませる
- 補助英語ラベルは主見出しより必ず弱くする
- Monospace は数値やIDに限定する

---

## 5. レイアウトの基本原則

### 5.1 共通原則
- 余白で階層を作る
- 枠線と背景差は最小限で使う
- 強調は色と余白で行い、装飾に頼らない
- カードは増やしすぎない
- 一度に全部を主張させない

### 5.2 情報の優先順位
どの画面でも、次の優先順位を守る。

1. 画面の主要目的
2. その場で読むべき一次情報
3. 補助情報
4. 将来機能や拡張領域
5. 関連リソース
6. スポンサー / アフィリエイト

---

## 6. BioFile Guide for Structure の画面原則

BioFile Guide は「構造ファイル入口の読解・トリアージ画面」である。  
可視化ダッシュボードではなく、最初の読み取りと仕分けを行うための画面として設計する。

### 6.1 画面構成
優先度順に、次の構成を基本とする。

1. 左側: 入力カラム
2. 右側上部: メイントリアージ領域
3. 右側中部: 拡張機能モジュール
4. 右側下部: 関連リソースとツール

### 6.2 入力カラム
入力カラムは安定的で控えめな見た目にする。

要件:
- 入力欄は静かで明快
- ローカルアップロード領域は過度に強調しない
- プライバシー注記は必須
- 「ブラウザ内に留まる」ことを落ち着いて伝える

### 6.3 メインレポート領域
メインレポートは3つの必須セクションで構成する。

#### 1. ファイルの概観
役割:
- 何者かを最初に伝える

表示方針:
- クリーンな要約行
- ID、フォーマット、実験手法、残基数などを整列
- 技術値は必要に応じて monospace を使う

#### 2. 注意事項
役割:
- 次に進む前の最初の注意点を伝える

最重要制約:
- 自由記述のAI診断カードにしない
- グラフ、円グラフ、スコア、信頼度メーターにしない
- 固定テンプレートの優先箇条書きで表現する
- メタ情報はチップや静かなサブ行に留める

#### 3. 推奨される次のアクション
役割:
- 次に何をすべきかを案内する

表示方針:
- 巨大ボタンにしない
- 「落ち着いた案内パネル」のリストにする
- 推奨行動は主役、技術コードは補助情報

### 6.4 拡張機能モジュール
将来機能は主役にしない。  
第1段階では、控えめなプレースホルダーまたはコンパクトスロットとして置く。

### 6.5 関連リソース
リソースは必ずメイントリアージ領域より下に置く。  
入力と結果の間、または注意事項の中に混ぜてはならない。

---

## 7. BioFile Guide における厳格な維持事項

次はデザイン変更後も必ず維持する。

- 3カード責務
- 注意事項の固定テンプレート箇条書き
- `resolved_identifier` 注意
- 外部リンクの外部サイト表示
- raw contract / 技術詳細の監査可能性
- warning priority / top3 / 折りたたみ
- next action の主役性
- 公式導線とスポンサー導線の視覚差

---

## 8. Portal の画面原則

Portal は単一製品のLPではなく、成長する科学ツール群の入口である。

### 8.1 必須セクション
1. トップナビゲーション / ブランド
2. 運営原則
3. 製品群
4. 研究ナビゲーションマップ
5. 参考資料

### 8.2 運営原則
企業アピールではなく、エコシステムの作動原理を伝える。

例:
- ブラウザネイティブ
- プライバシー優先
- 科学的証拠第一

### 8.3 製品群
製品群は、単なるカードの羅列ではなく、カテゴリと成熟度を伴って整理する。

推奨カテゴリ例:
- Read / Guide
- Analyze / Diagnose
- Refine / Simulate

成熟度例:
- Stable
- Beta
- Prototype

### 8.4 ワークフロー
研究プロセスの中で、各製品がどこに位置するかを示す。  
単なる機能一覧ではなく、研究文脈の地図として扱う。

### 8.5 参考資料
Portal の参考資料も、BioFile Guide と同じく厳格な階層差を持たせる。

## 9. 公式 / 厳選 / スポンサーの3階層ルール

これらは必ず同列扱いしない。

### 9.1 公式・公的リソース
- 最も中立的
- 最も信頼性が高い
- 最もクリーンで静かな扱い

### 9.2 厳選された推奨ツール
- 中間層
- 編集の意図が感じられる
- 少し温かみがある
- ただし派手にしない

### 9.3 スポンサー / アフィリエイト
- 最も優先度が低い
- `SPONSORED` / `AFFILIATE` ラベル必須
- マーケティングコピーを避ける
- 文脈説明は短く
- バナー広告のようにしない

---

## 10. バッジとラベル

### 10.1 成熟度ラベル
Portal では成熟度ラベルを使い、階層を作る。

- Stable
- Beta
- Prototype

### 10.2 外部リンクラベル
BioFile Guide では外部リンクに必ず外部サイト表示を付ける。  
リンク本体とラベルは同一まとまりで扱う。

### 10.3 スポンサーラベル
スポンサー / アフィリエイトには明示ラベルを必須とする。  
見落とし防止のためだが、過剰に主張させない。

---

## 11. 第1段階でやること / やらないこと

### 11.1 第1段階でやること
- 共通デザイントークン導入
- タイポ階層整理
- BioFile Guide の見た目整理
- Portal の情報設計整理
- リソース階層の視覚差導入

### 11.2 第1段階ではやらないこと
- 派手なアニメーション
- ダークモード
- 完全なレスポンシブ最適化の深掘り
- モバイル専用最適化の作り込み
- Mol* / iCn3D 埋め込み前提の大レイアウト改変
- アイコンシステム大刷新
- デザインシステムの過剰抽象化

---

## 12. ネガティブ制約

以下は禁止とする。

- 巨大CTAボタン中心のSaaS表現
- 派手なローンチ演出
- ネオン / グロウ / 強いグラデーション
- サイバーパンク風表現
- 桜、鳥居、筆文字などの直接的な和風演出
- 注意事項のグラフ化、ダッシュボード化
- 公式リソースとスポンサーを同じ重みで並べること
- すべてが一度に主張する過密レイアウト

---

## 13. 実装上の優先順位

実装では、次の順で適用する。

1. 共通デザイントークン
2. タイポグラフィ階層
3. BioFile Guide の3カードまわり
4. 関連リソース階層
5. Portal の構造整理
6. 第2段階以降の洗練
