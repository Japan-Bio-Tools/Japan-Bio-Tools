# BioFile Guide for Structure

## 設計書

---

## この文書の位置づけ

この文書は、BioFile Guide for Structure の**親正本**である。
実装エージェントは、初手でこの文書を読み、詳細実装へ進む前に不変条件、参照ルール、詳細正本の所在を確認する。

本プロダクトは、構造解析サービスでも、独自ビューアでも、総合変換ツールでもない。
位置づけは一貫して、**構造ファイル入口翻訳機**である。

この親正本は、詳細本文を大量に再掲しない。
詳細定義の正本は [`architecture/`](architecture/) 配下の各子文書に置く。
親正本は、原則、主権、参照先、初期実装の入口を定義する。

---

## 先に固定するもの

本プロダクトで最も重要なのは、画面や実装都合ではなく、次の契約を先に固定することである。

* 何を判定対象とし、何を判定対象外とするか
* どの条件で何を返すか
* どの条件で unknown に落とすか
* success と error をどう出し分けるか
* どの外部ソースをどう使い、競合時にどう扱うか
* どの URL をどう安全に組み立てるか
* 結論をどう見せ、根拠をどう添えるか
* 何をもって品質とみなすか

これらの詳細正本は、下記の「詳細正本一覧」に従う。

## 2. プロダクト定義

### 2-1. 最小価値

BioFile Guide for Structure は、入力された構造データに対して、最低限次の3点を返す。

1. この入力は何者か
2. 最初に気をつけることは何か
3. 次にどこを開くべきか

この3点が、本プロダクトの最小価値である。

### 2-2. 設計原則

本プロダクトは以下を設計原則とする。

* 一次判定はルールベースで成立させる
* ID入力では公式APIを積極利用する
* classification engine は外部API生レスポンスに直接依存しない
* 結論だけでなく根拠を返す
* 分からないものを無理に断定しない
* unknown は失敗ではなく正しい出力として扱う
* 深掘りは外部基盤へ橋渡しする
* ローカルファイルは原則ブラウザ内処理とする
* 保存しないことを信頼の土台に置く
* 失敗時にも次の行動を返す
* UI は契約を表示する器であり、契約をねじ曲げない

### 2-3. 非目的

本プロダクトは以下を目的としない。

* 公式検索ポータルの代替
* 公式ビューアの代替
* 高度な構造解析
* 妥当性評価の保証
* 構造比較や重ね合わせ
* 変換・編集の総合ツール化
* 構造本文のサーバー保存
* アカウント前提の履歴管理

## 親正本が保持する不変条件

* BioFile Guide for Structure は、入力された構造データについて「何者か」「最初に気をつけること」「次にどこを開くべきか」を返す。
* 一次判定はルールベースで成立させる。
* ID入力では公式APIを積極利用する。
* ローカルファイルは原則ブラウザ内処理とし、構造本文、座標、ファイル名、生テキストを外部送信しない。
* classification engine は外部API生レスポンスに直接依存せず、正規化済みDTOだけを受け取る。
* 分からないものを無理に断定しない。
* unknown は失敗ではなく正しい出力として扱う。
* 失敗時にも次の行動を返す。
* UI は契約を表示する器であり、契約をねじ曲げない。
* Mol* は構造読込・表示・ローカル閲覧導線の基盤として使ってよいが、判定責務を負わせない。
* `resolved_identifier` は URL 組み立てと API 問い合わせに使う正規化済み canonical identifier であり、実在確認済みであることを意味しない。
* `next_links.href` は allowlist 済み URL テンプレートと `resolved_identifier` からのみ生成する。
* 匿名計測に識別子、ファイル名、入力本文、派生構造情報を含めない。

## 実装エージェントの初期読書順

1. この親正本で、不変条件と詳細正本の所在を確認する。
2. [実装手順書](architecture/implementation-guide.md) で、初期コーディング順序と DoD を確認する。
3. [詳細設計書](architecture/detailed-design.md) で、DTO、応答スキーマ、adapter、merge strategy、classification engine、builder、formatter の境界を確認する。
4. [判定表詳細](architecture/decision-table.md) で、入力判定、分類、unknown / error 分岐を実装する。
5. [reason / evidence code 定義書](architecture/reason-evidence-codes.md)、[unknown UI 文言仕様](architecture/unknown-ui.md)、[next_links 選定仕様](architecture/next-links.md) を参照して、根拠、表示、次の一手を契約どおりに接続する。
6. [匿名計測仕様](architecture/anonymous-telemetry.md) と [ゴールドセット定義書](architecture/gold-set.md) で、計測と検証を契約違反なく確認する。

## 子文書の責務境界

* 詳細設計書は、実装上の責務分割と応答構造を定義する。判定表、UI文言、next_links、匿名計測、ゴールドセットの詳細正本にはしない。
* 判定表詳細は、条件と返却値の対応表を定義する。文言テンプレートや URL テンプレートの詳細正本にはしない。
* reason / evidence code 定義書は、コードと根拠付けの正本である。UI 表示順やリンク選定の正本にはしない。
* unknown UI 文言仕様は、分からない時に無理に断定しない表示契約を定義する。分類ロジックの正本にはしない。
* next_links 選定仕様は、次の一手と URL 生成の正本である。分類ロジックや文言生成の正本にはしない。
* 匿名計測仕様は、計測してよいもの、してはならないものを定義する。入力本文や識別子の送信を許可する文書ではない。
* ゴールドセット定義書は、品質確認と受け入れ観点を定義する。新機能ロードマップではない。
* 実装手順書は、初期コーディングの順序を示す。長期構想や仕様追加のロードマップではない。

## 最低限の全体要約

本設計体系の核心は、画面ではない。
**判定契約である。**

このプロダクトの価値は、全部分かることではない。
価値は次の4点にある。

* 何者かを短く返す
* 注意点を先に示す
* 分からないときは無理に断定しない
* 次にどこを見れば前に進めるかを返す

その「次の一手」は、公式サービス群と競合するためではなく、**それらを正しく使えるようにするための翻訳結果**である。
本プロダクトは **「APIで判断し、Mol* で見せる」** という分担で成立させる。

この原則を壊さない限り、BioFile Guide for Structure は、MVPでありながら崩れにくい土台を持った入口翻訳機として実装へ進める。
