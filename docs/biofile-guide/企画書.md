# BioFile Guide for Structure

## 基本設計書

---

## 0. 本書の目的

本書は、**BioFile Guide for Structure** の基本設計を定義する文書である。

本プロダクトは、構造解析サービスでも、独自ビューアでも、総合変換ツールでもない。
位置づけは一貫して、**構造ファイル入口翻訳機**である。

したがって基本設計において最も重要なのは、UIの見た目や実装都合ではなく、次の契約を先に固定することである。

* 何を判定対象とし、何を判定対象外とするか
* どの条件で何を返すか
* どの条件で `unknown` に落とすか
* 結論をどう見せ、根拠をどう添えるか
* 次にどこへ誘導するか
* 何を計測し、何を計測しないか
* 何をもって品質とみなすか

本書は、企画書を実装可能なレベルへ降ろすための**基本設計上の正本**とする。

---

## 1. プロダクトの設計原則

### 1-1. プロダクト定義

BioFile Guide for Structure は、入力された構造データに対して、以下の3点を返す。

1. **このファイルは何者か**
2. **最初に気をつけることは何か**
3. **次にどこを開くべきか**

この3点が、本プロダクトの最小価値である。

### 1-2. 設計原則

本プロダクトは以下を設計原則とする。

* 一次判定はルールベースで成立させる
* 結論だけでなく根拠を返す
* 分からないものを無理に断定しない
* `unknown` は失敗ではなく正しい出力として扱う
* 低根拠の強断定を避ける
* 深掘りは外部基盤へ橋渡しする
* ローカルファイルは原則ブラウザ内処理とする
* 保存しないことを信頼の土台に置く

### 1-3. 非目的

本プロダクトは以下を目的としない。

* 高度な構造解析
* 妥当性評価の保証
* 独自ビューアの提供
* ファイル変換や編集の総合機能
* 長文AI相談の提供
* アカウント前提の保存・履歴管理
* 構造データ本文の収集・蓄積

## 2. 判定対象と非対象

### 2-1. 判定対象

MVP時点の判定対象は以下に限定する。

* 4文字PDB ID
* 拡張PDB ID
* ローカルPDBファイル
* ローカルmmCIFファイル

### 2-2. 判定対象で返す情報

判定対象に対して返す情報は以下とする。

* `input_type`
* `file_format`
* `record_type`
* `source_database`
* `experiment_method`
* `model_count`
* `chain_count`
* `ligand_present`
* `water_present`
* `legacy_pdb_compatibility`
* `legacy_pdb_reason_code`
* `legacy_pdb_reason_text`
* `confidence`
* `evidence`
* `beginner_warning`
* `recommended_next_step`
* `next_links`

### 2-3. 判定対象外

以下はMVPの判定対象外とする。

* 変換要求
* chain抽出
* water除去
* backbone-only 出力
* 構造比較
* 変異影響推定
* 分子間相互作用解析
* 妥当性スコア計算
* 独自レンダリング
* サーバー保存前提の履歴管理

### 2-4. 判定不能と対象外の違い

設計上、**判定不能**と**対象外**は明確に分ける。

* 判定不能: 対象入力ではあるが、根拠不足のため `unknown` を返す状態
* 対象外: そもそも本プロダクトが扱わない要求や機能

## 3. 入出力の基本設計

### 3-1. 入力種別

`input_type` は以下のいずれかとする。

* `pdb_id`
* `extended_pdb_id`
* `local_pdb`
* `local_mmcif`

### 3-2. 出力スキーマ

MVPの返却構造は以下を基本契約とする。

```json id="jvd5g4"
{
  "input_type": "pdb_id | extended_pdb_id | local_pdb | local_mmcif",
  "file_format": "pdb | mmcif | unknown",
  "record_type": "experimental_structure | computed_model | integrative_structure | unknown",
  "source_database": "PDB | AlphaFoldDB | ModelArchive | local_file | unknown",
  "experiment_method": "string | null",
  "model_count": 0,
  "chain_count": 0,
  "ligand_present": true,
  "water_present": true,
  "legacy_pdb_compatibility": "compatible | caution | incompatible | unknown",
  "legacy_pdb_reason_code": "string | null",
  "legacy_pdb_reason_text": "string | null",
  "confidence": "high | medium | low",
  "evidence": [
    {
      "code": "string",
      "detail": "string"
    }
  ],
  "beginner_warning": [
    "string"
  ],
  "recommended_next_step": "string",
  "next_links": [
    {
      "label": "RCSB | PDBe | PDBj | Mol* | iCn3D",
      "reason": "string",
      "url": "string"
    }
  ]
}
```

### 3-3. 出力整合性ルール

以下を出力整合性ルールとする。

* `confidence` だけが高く、`evidence` が空である状態は禁止
* `record_type` が断定されているのに `recommended_next_step` が空である状態は禁止
* `unknown` の場合でも `recommended_next_step` は必須
* `legacy_pdb_compatibility` が `caution` 以上なら理由文を必須にする
* `next_links` は0件不可。最低1件返す

## 4. 判定表

本章は、本プロダクトの中心契約である。
実装は必ずこの判定表に従う。

---

### 4-1. `file_format` 判定表

| 条件                | 判定        |
| ----------------- | --------- |
| PDB形式の明確な構文を満たす   | `pdb`     |
| mmCIF形式の明確な構文を満たす | `mmcif`   |
| 判定不能              | `unknown` |

補足:

* `pdb_id` 入力は原則 `pdb`
* `extended_pdb_id` 入力は原則 `mmcif`
* ローカル入力は実ファイル構文を優先して判定する

---

### 4-2. `record_type` 判定表

| 優先順位 | 条件                                       | 判定                       |
| ---- | ---------------------------------------- | ------------------------ |
| 1    | 実験構造を直接示す明示メタデータあり                       | `experimental_structure` |
| 1    | ModelCIF系など計算モデルを直接示す明示メタデータあり           | `computed_model`         |
| 1    | IHM系など integrative/hybrid を直接示す明示メタデータあり | `integrative_structure`  |
| 2    | 入力IDや由来情報から強く一意推定できる                     | 対応する値                    |
| 3    | 弱い状況証拠のみ                                 | 原則 `unknown`             |
| -    | 根拠競合・根拠不足                                | `unknown`                |

設計原則:

* 直接根拠を最優先する
* 弱い推定だけで断定しない
* `computed_model` と `integrative_structure` の取り違えを避けるため、曖昧な場合は `unknown`

---

### 4-3. `source_database` 判定表

| 条件                                   | 判定             |
| ------------------------------------ | -------------- |
| 4文字PDB ID 入力                         | `PDB`          |
| 拡張PDB ID 入力                          | `PDB`          |
| AlphaFoldDB 由来を直接示す強い provenance あり  | `AlphaFoldDB`  |
| ModelArchive 由来を直接示す強い provenance あり | `ModelArchive` |
| ローカルファイルで強い provenance なし            | `local_file`   |
| 判定不能                                 | `unknown`      |

設計原則:

* ローカルファイルは**外見だけで外部DBを断定しない**
* `local_file` は消極値ではなく、保守的で正しい値として扱う
* “それっぽい”だけでは `AlphaFoldDB` や `ModelArchive` を返さない

---

### 4-4. `legacy_pdb_compatibility` 判定表

| 条件                                           | 判定             |
| -------------------------------------------- | -------------- |
| 旧PDB形式で安全に扱える根拠が十分ある                         | `compatible`   |
| 情報欠落リスクがあるが断定不能                              | `caution`      |
| 拡張PDB ID前提、integrative、mmCIF前提表現など旧PDB非互換が明確 | `incompatible` |
| 根拠不足                                         | `unknown`      |

設計原則:

* yes/noでは返さない
* “落とせるか”ではなく、“落としたとき危険か”を返す
* 慎重側に倒す

---

### 4-5. `model_count` 判定表

| 条件       | 値  |
| -------- | -- |
| 明確に取得可能  | 実数 |
| 未取得・判定不能 | 0  |

補足:

* `0` は「モデルがない」ではなく「未取得/未判定」を含む
* UIでは `0` をそのまま見せず、未判定表示へ変換してよい

---

### 4-6. `chain_count` 判定表

| 条件       | 値  |
| -------- | -- |
| 明確に取得可能  | 実数 |
| 未取得・判定不能 | 0  |

補足:

* `chain_count` も `0` を内部上の未判定として許容する
* UIでは「未確認」等に言い換えてよい

---

### 4-7. `ligand_present` / `water_present` 判定表

| 条件   | 値       |
| ---- | ------- |
| 検出あり | `true`  |
| 検出なし | `false` |

補足:

* ローカル構文解析で検出不能な場合は、MVPでは false ではなく未判定扱いにすべき可能性がある
* ただしMVPスキーマを簡素化するため、内部では未判定フラグを持ち、UIで吸収する設計を推奨する

---

## 5. reason code / evidence code 一覧

---

### 5-1. `legacy_pdb_reason_code`

初期コードは以下を基本とする。

| code                               | 意味                                |
| ---------------------------------- | --------------------------------- |
| `extended_id_requires_mmcif`       | 拡張PDB ID前提でmmCIF利用が必要             |
| `integrative_not_supported_in_pdb` | integrative/hybrid表現は旧PDBで安全に扱えない |
| `mmcif_only_representation`        | mmCIF前提の表現要素を含む                   |
| `size_or_schema_risk`              | サイズまたはスキーマ上の欠落リスクあり               |
| `unknown_origin`                   | 出自や内容が不十分で安全性判断不可                 |
| `null`                             | 理由不要                              |

---

### 5-2. `evidence.code`

初期コードは以下を基本とする。

| code                                     | 意味               |
| ---------------------------------------- | ---------------- |
| `explicit_exptl_method`                  | 実験法を直接示す情報あり     |
| `explicit_modelcif_marker`               | ModelCIF系の直接根拠あり |
| `explicit_ihm_marker`                    | IHM系の直接根拠あり      |
| `pdb_identifier_detected`                | PDB ID を検出       |
| `extended_pdb_identifier_detected`       | 拡張PDB ID を検出     |
| `multiple_models_detected`               | 複数モデルを検出         |
| `single_model_detected`                  | 単一モデルを検出         |
| `multiple_chains_detected`               | 複数鎖を検出           |
| `water_records_detected`                 | water を検出        |
| `ligand_records_detected`                | ligand を検出       |
| `local_file_without_reliable_provenance` | ローカルファイルで出自断定不可  |
| `legacy_pdb_incompatibility_marker`      | 旧PDB非互換の強い兆候あり   |
| `legacy_pdb_caution_marker`              | 旧PDB注意の兆候あり      |
| `format_pdb_detected`                    | PDB形式を検出         |
| `format_mmcif_detected`                  | mmCIF形式を検出       |

---

### 5-3. evidence の必須ルール

* `record_type` を `unknown` 以外で返す場合、対応する直接または強い evidence を最低1件必須とする
* `legacy_pdb_compatibility` を `incompatible` にする場合、理由コードと evidence の両方を必須とする
* `source_database` を `AlphaFoldDB` / `ModelArchive` と返す場合、強い provenance evidence を必須とする

---

## 6. unknown と confidence の表示契約

---

### 6-1. 基本原則

`unknown` は失敗ではない。
誤断定を避けるための正しい出力である。

`confidence` は連続値ではなく、外向きには以下の3段階のみを使用する。

* `high`
* `medium`
* `low`

---

### 6-2. confidence 判定契約

| confidence | 条件                    |
| ---------- | --------------------- |
| `high`     | 明示的で直接的な根拠が複数ある       |
| `medium`   | 明示的根拠が1つ、または間接根拠が複数ある |
| `low`      | 間接根拠のみ、または根拠が弱い       |

設計ルール:

* `low` で強い断定をしない
* `low` のときは `unknown` へのフォールバックを優先する
* `high` は evidence の質で決め、件数だけで決めない

---

### 6-3. unknown 表示契約

`unknown` 表示時は、以下を必須とする。

* 断定できない理由
* それでも確認できた事実
* 次に確認すべき場所
* ユーザーを前に進める文面

禁止事項:

* 「不明です」だけで終わる
* 根拠が弱いのに断定口調に寄せる
* 原典確認先を返さない

---

### 6-4. unknown時の標準文面方針

表示文面は以下の構造を持つ。

1. 断定不能の明示
2. 断定不能の理由
3. 確認済み情報
4. 次の一手

例:

* 「このファイルの由来は、この情報だけでは断定できません。」
* 「ただし、mmCIF形式であることと、複数モデルを含むことは確認できています。」
* 「まずは RCSB または PDBe の原典情報を確認してください。」

---

## 7. next_links 選定契約

`next_links` は単なるリンク集ではなく、**次に何を確認すべきかの翻訳結果**とする。

---

### 7-1. 選定原則

| 目的                 | 優先リンク                  |
| ------------------ | ---------------------- |
| 原典確認・基本情報確認        | `RCSB`, `PDBe`, `PDBj` |
| まず構造を眺めたい          | `Mol*`                 |
| 3D/配列/相互作用を横断して見たい | `iCn3D`                |

---

### 7-2. 選定ルール

* `recommended_next_step` は最上位の一手を1つ返す
* `next_links` はその一手を支える複数候補を返してよい
* 原典確認が必要なケースでは、可視化リンクより原典リンクを優先する
* `unknown` の場合は原典確認リンクを優先する
* 可視化だけで解決しない問いに対して、Mol* を第一候補にしすぎない

---

### 7-3. 最低保証

* すべての結果で最低1件の `next_links` を返す
* `unknown` の場合は最低1件の原典確認系リンクを含める

---

## 8. 匿名計測契約

---

### 8-1. 基本方針

本プロダクトは保存しない設計を原則とする。
匿名計測を導入する場合も、その原則を壊してはならない。

### 8-2. 送信可能なもの

送信可能な情報は、原則として以下に限定する。

* 画面到達イベント
* ボタン/リンククリックイベント
* `unknown` 返却イベント
* `recommended_next_step` クリックイベント
* 記事→ツール導線イベント
* エラー種別コード

### 8-3. 送信禁止

以下は送信禁止とする。

* ローカルファイル本文
* 座標情報
* 原ファイル名
* ファイル中の可逆的な内容
* 生の構造テキスト
* 個別の分子情報を復元可能なデータ

### 8-4. event code 設計

匿名計測はイベントコード中心で実装する。
例:

* `result_rendered`
* `result_unknown_rendered`
* `click_recommended_next_step`
* `click_link_rcsb`
* `click_link_pdbe`
* `click_link_molstar`
* `click_link_icn3d`
* `error_parse_failed`

## 9. ゴールドセット方針

---

### 9-1. 目的

ゴールドセットは、判定品質を主観ではなく実測で検証するための基準集合である。

### 9-2. 初期件数

初期は 20〜30 件を基準とする。
将来的には 30〜50 件への拡張を想定する。

### 9-3. 構成要件

最低限、以下を含む。

* 4文字PDB ID
* 拡張PDB ID
* ローカルPDB
* ローカルmmCIF
* 実験構造
* 予測構造
* integrative structure
* 複数モデル
* 複数鎖
* ligand 含有
* water 含有
* `unknown` が正答になる曖昧ケース

### 9-4. 検証項目

ゴールドセットでは少なくとも以下を確認する。

* `record_type` 一致率
* `source_database` の安全性
* `legacy_pdb_compatibility` の妥当性
* `recommended_next_step` の妥当性
* `unknown` を返すべきケースで誤断定していないか

### 9-5. 品質哲学

本プロダクトは、
**当てること** と同じ重さで、
**無理に当てにいかないこと** を品質とみなす。

## 10. 画面設計

ここから画面設計に入る。
ただし、画面は前章までの契約を表示する器であり、先に契約を曲げてはならない。

---

### 10-1. 画面一覧

MVPで必要な画面は以下とする。

* 入力画面
* 判定結果画面
* エラー/対象外案内領域
* 利用上の注意・免責表示
* プライバシー/計測方針表示

---

### 10-2. 入力画面

入力画面では以下を受け付ける。

* PDB ID 入力
* 拡張PDB ID 入力
* ローカルPDBファイル選択
* ローカルmmCIFファイル選択

表示要件:

* 対応入力を明示する
* 保存しないことを近接表示する
* 深い解析はしないことを明示する

---

### 10-3. 結果画面

結果画面は3カード固定とする。

#### カード1: このファイルは何者か

表示候補:

* `record_type`
* `file_format`
* `source_database`
* `experiment_method`
* `confidence`

#### カード2: 最初に気をつけること

表示候補:

* `beginner_warning`
* `legacy_pdb_compatibility`
* `legacy_pdb_reason_text`
* `model_count`
* `chain_count`
* `ligand_present`
* `water_present`

#### カード3: 次に開く場所

表示候補:

* `recommended_next_step`
* `next_links`

---

### 10-4. 根拠表示

根拠は `evidence` として補助領域に表示する。

表示ルール:

* 初期状態は簡潔表示
* 必要に応じて展開可能
* ユーザー向けには専門用語を補足する
* evidence code をそのまま見せる場合も、説明文を併記する

---

### 10-5. unknown時の画面特例

`unknown` の場合は以下を追加する。

* 断定不能の理由の明示
* 原典確認先の強調
* 「分からないまま放置しない」導線の明示

---

## 11. 画面遷移

### 11-1. 基本遷移

1. 入力
2. 判定処理
3. 結果表示
4. 外部リンク遷移

### 11-2. 例外遷移

* 構文解析失敗 → エラー案内
* 対象外要求 → 対象外案内
* 判定不能 → `unknown` 結果表示

---

## 12. エラー設計

### 12-1. エラー分類

* `parse_failed`
* `unsupported_input`
* `empty_input`
* `invalid_identifier`
* `unknown_classification`

### 12-2. エラー表示方針

* ユーザーを責めない
* 何ができなかったかを短く示す
* 次にできる行動を返す

---

## 13. 非機能設計

### 13-1. プライバシー

* ローカルファイルは原則ブラウザ内処理
* サーバー保存しない
* 学習利用しない
* 再配布しない

### 13-2. 信頼性

* `unknown` を正式出力として扱う
* 断定のしすぎを避ける
* 原典確認導線を常に残す

### 13-3. 保守性

* 判定表、reason code、evidence code、next_links ルールは分離管理可能にする
* UI文面と判定ロジックを直接結合しすぎない

### 13-4. 拡張性

将来拡張時も、以下の順序を守る。

1. 判定精度改善
2. 根拠表示改善
3. next_links 改善
4. ゴールドセット拡張
5. 軽い比較や補助機能の検討

---

## 14. 実装上の分割方針

実装は少なくとも以下の責務に分割する。

* 入力判定
* 構文判定

## 15. 受け入れ条件

本基本設計に対するMVP受け入れ条件は以下とする。

* 判定対象/非対象が実装で守られている
* `record_type` / `source_database` / `legacy_pdb_compatibility` が判定表どおりに返る
* `unknown` と `confidence` の表示契約が守られている
* `evidence` が空の断定結果を返さない
* `next_links` が役割に沿って返る
* 匿名計測契約に違反しない
* 初期ゴールドセットで検証可能である
* 3カードUIが契約を壊さず表示できる

## 17. 最終確認

本基本設計の核心は、画面ではない。
**判定契約である。**

このプロダクトの価値は、
「全部分かる」ことではなく、

* 何者かを短く返し
* 注意点を先に示し
* 分からない時は無理に断定せず
* 次にどこを見れば前に進めるかを返す

ことにある。

この原則を壊さない限り、BioFile Guide for Structure は企画書の勝ち筋を保ったまま実装へ進める。
